# เว็บติดตามใบกำกับภาษีที่ยังไม่ได้รับ — BENZ

ระบบสำหรับทีมภายใน ใช้บันทึก/ติดตามรายการซื้อที่ยังไม่ได้รับใบกำกับภาษีจากผู้ขาย พร้อมวันที่คาดว่าจะได้รับ
สถานะ และ aging (ค้างนานแค่ไหน) — ทุกคนที่ login แล้วมีสิทธิ์เข้าถึงข้อมูลเท่ากัน (equal access)

Stack: Next.js 16 (App Router, client-heavy) + Supabase (database + auth) + Tailwind CSS
ออกแบบให้ deploy บน Vercel

## 3. Deploy ขึ้น Vercel

วิธีที่แนะนำ (ไม่ต้องแชร์ API token ให้ใคร):

1. สร้าง repository บน GitHub แล้ว push โค้ดโปรเจกต์นี้ขึ้นไป:
   ```bash
   git init
   git add .
   git commit -m "Initial commit"
   git branch -M main
   git remote add origin <URL ของ repo คุณ>
   git push -u origin main
   ```
2. เข้า [vercel.com/new](https://vercel.com/new) แล้วกด **Import** repository ที่เพิ่ง push ขึ้นไป
   — Vercel จะตรวจพบว่าเป็น Next.js โปรเจกต์เองอัตโนมัติ ไม่ต้องตั้งค่า build command เพิ่ม
3. ก่อนกด Deploy ให้เพิ่ม **Environment Variables** ในหน้าตั้งค่าของ Vercel:
   | Name | Value |
   |---|---|
   | `NEXT_PUBLIC_SUPABASE_URL` | Project URL จาก Supabase (ขั้นตอนที่ 1) |
   | `NEXT_PUBLIC_SUPABASE_ANON_KEY` | anon/public key จาก Supabase (ขั้นตอนที่ 1) |
4. กด **Deploy** — ครั้งต่อไปที่ push โค้ดขึ้น branch `main` จะ deploy ให้อัตโนมัติทุกครั้ง

## 6. นำเข้ารายการจาก Excel

หน้า **"บันทึกค่าใช้จ่าย"** มีปุ่ม **"นำเข้าจาก Excel"** ข้าง ๆ ปุ่ม "+ เพิ่มรายการ" สำหรับบันทึกหลาย
รายการพร้อมกันโดยไม่ต้องกรอกทีละรายการผ่านฟอร์ม (กดปุ่มลัด "Import Excel" ใน Quick Actions ของหน้า
Dashboard ภาพรวมก็เปิดแผงนี้ให้ทันทีเช่นกัน — ดูหัวข้อ 7)

ขั้นตอนใช้งาน:

1. กด **"ดาวน์โหลดเทมเพลต Excel"** ในแผงนำเข้า — ได้ไฟล์ `.xlsx` ที่มีหัวคอลัมน์ถูกต้องพร้อมแถวตัวอย่าง 1 แถว
2. กรอกข้อมูลต่อจากแถวตัวอย่าง (หรือลบแถวตัวอย่างทิ้งแล้วกรอกของจริงแทน) แล้วบันทึกไฟล์
3. กด **"เลือกไฟล์ Excel..."** เลือกไฟล์ที่กรอกแล้ว — ระบบจะตรวจสอบทุกแถวและแสดงตารางตัวอย่างทันที
   พร้อมสถานะรายแถว (พร้อมนำเข้า / มีปัญหา พร้อมเหตุผล)
4. ตรวจสอบตัวอย่างแล้วกด **"นำเข้า N รายการ"** — จะบันทึกเฉพาะแถวที่ผ่านการตรวจสอบเท่านั้น
   (แถวที่มีปัญหาจะถูกข้ามไป ไม่ต้องแก้ทุกแถวให้ถูกก่อนถึงจะนำเข้าแถวอื่นได้)

หัวคอลัมน์ในเทมเพลต:

| คอลัมน์ | จำเป็นไหม | หมายเหตุ |
|---|---|---|
| ผู้ขาย | จำเป็น | |
| วันที่ทำรายการ | จำเป็น | รองรับเซลล์วันที่จริงของ Excel, `YYYY-MM-DD`, หรือ `DD/MM/YYYY` |
| รายละเอียด | ไม่จำเป็น | |
| ยอดก่อน VAT | จำเป็น | ต้องเป็นตัวเลขมากกว่า 0 (ใส่ comma คั่นหลักพันได้ เช่น `1,000`) |
| VAT | ไม่จำเป็น | **เว้นว่างไว้ระบบจะคำนวณให้อัตโนมัติ 7%** จากยอดก่อน VAT (เหมือนฟอร์มเพิ่มรายการทีละตัว) |
| เลขที่อ้างอิง | ไม่จำเป็น | |
| วันที่คาดว่าจะได้รับใบกำกับภาษี | ไม่จำเป็น | ถ้ากรอกต้องไม่ก่อนวันที่ทำรายการ |
| หมายเหตุ | ไม่จำเป็น | |

แถวที่ว่างทั้งแถว (เช่นแถวว่างท้ายไฟล์) จะถูกข้ามไปอัตโนมัติ ไม่ถือเป็นปัญหา

> **หมายเหตุด้านเทคนิค:** ใช้ไลบรารี `xlsx` (SheetJS) อ่าน/เขียนไฟล์ Excel ฝั่ง client ทั้งหมด — ไม่มีการ
> ส่งไฟล์ไปประมวลผลที่ server ใดๆ `npm audit` แจ้งช่องโหว่ระดับ high ในไลบรารีนี้ (prototype pollution,
> ยังไม่มี patch) แต่ประเมินแล้วว่าความเสี่ยงต่ำสำหรับการใช้งานนี้ เพราะเป็นเครื่องมือใช้ภายใน
> ประมวลผลเฉพาะไฟล์ที่ผู้ใช้ในทีมอัปโหลดเอง และผลกระทบจำกัดอยู่แค่ browser tab ของผู้ใช้เอง
> (ไม่กระทบฐานข้อมูลหรือผู้ใช้คนอื่น) — ทางเลือกอื่น (`exceljs`) มี dependency tree ที่หนักกว่าและ
> deprecated มากกว่าโดยไม่ได้ลดความเสี่ยงจริงลงในบริบทนี้

## 7. เมนู Sidebar

หน้าเว็บมีแถบเมนูด้านซ้าย (Sidebar) แบบระบบ ERP ครอบคลุมทั้งเว็บ โครงสร้างเมนูปัจจุบัน (ปรับใหม่ล่าสุด
2026-07-15 แยกหน้าภาพรวมออกจากหน้าทำงานจริง ให้เหมาะกับการใช้งานระยะยาว):

- **🏠 Dashboard** — หน้าภาพรวมระบบ เป็น **หน้าแรกที่เห็นหลังล็อกอิน** (active เป็นค่าเริ่มต้นเสมอ)
  ดูหัวข้อ "Dashboard ภาพรวม" ด้านล่าง
- **📤 บันทึกการจ่ายเงิน** — บันทึกค่าใช้จ่าย (มีฟีเจอร์จริง), รายงานจ่ายเงิน (ยังไม่ทำ)
- **🔄 กระทบยอด** — Bank Reconcile (ยังไม่ทำ), **VAT Reconcile** (หมวดย่อยซ้อนอยู่ข้างใน ไม่ใช่หน้า
  เนื้อหาเอง): รายงานภาษีซื้อ (มีฟีเจอร์จริง — [หัวข้อ 8](#8-รายงานภาษีซื้อ-vat-reconcile)),
  รายงานภาษีขาย (ยังไม่ทำ — รอระบบบันทึกยอดขาย), ภาษีซื้อไม่ถึงกำหนด (ยังไม่ทำ), ตรวจสอบข้อมูล
  (ยังไม่ทำ)

โครงสร้างเมนูรองรับซ้อนได้ไม่จำกัดระดับ (`NavSection.children` เป็น `NavEntry[]` ทำให้หมวดหนึ่ง
มีหมวดย่อยซ้อนอยู่ข้างในได้ เช่น "VAT Reconcile" ที่ซ้อนอยู่ใน "กระทบยอด" ด้านบน) แก้ที่
`lib/navigation.ts` ที่เดียวถ้าต้องการเพิ่ม/ลบเมนู

เมนูย่อยที่ยังไม่มีฟีเจอร์จริง (`implemented: false`) คลิกได้ตามปกติแต่จะขึ้นหน้า **"เร็วๆ นี้"**
แทน (ไม่ใช่ปุ่มที่กดแล้วไม่มีอะไรเกิดขึ้น) — เพิ่มฟีเจอร์จริงทีหลังได้โดยไม่ต้องแก้ Sidebar เลย
แค่สร้างหน้า/component ใหม่ เปลี่ยน `implemented: false` เป็น `true` ที่ `lib/navigation.ts` แล้วเพิ่ม
case ให้ id นั้นใน `renderActiveContent()` ที่ `app/dashboard/page.tsx`

### Dashboard ภาพรวม

หน้าแรกของระบบ เป็นหน้า **"อ่านอย่างเดียว"** ล้วน ๆ — ไม่มีฟอร์ม ปุ่มแก้ไข หรือปุ่มลบข้อมูลใด ๆ อยู่ใน
หน้านี้เอง (ทุกอย่างที่ต้องแก้ไขข้อมูลจริงพากลับไปหน้า "บันทึกค่าใช้จ่าย" เสมอ) ประกอบด้วย:

1. **KPI Cards** 5 ใบ (รอรับใบกำกับภาษี, ยอดรวมที่รอรับ, VAT ที่รอรับ, เกินกำหนด, ได้รับแล้ว) — ย้ายมาจาก
   หน้า "บันทึกค่าใช้จ่าย" เดิมทั้งหมด (component `StatsCards` ตัวเดียวกัน ไม่ได้แก้ logic การคำนวณเลย)
2. **ตารางรายการรอรับใบกำกับภาษีล่าสุด** — 5 รายการล่าสุด (เรียงจากรายการที่เพิ่งบันทึกเข้าระบบก่อน)
3. **รายการเกินกำหนด** — 5 รายการที่เกินกำหนดนานสุดก่อน
4. **สรุป VAT รายเดือน** — ย้ายมาจากหน้า "บันทึกค่าใช้จ่าย" เดิมเช่นกัน (component `MonthlyVatSummary`
   ตัวเดียวกัน)
5. **Quick Actions** — ปุ่มลัด 4 ปุ่ม: เพิ่มค่าใช้จ่าย, Import Excel, รายงานภาษีซื้อ, รายงานภาษีขาย

ทุกการ์ด/ปุ่มในหน้านี้กดแล้วพาไปเมนูที่เกี่ยวข้องเสมอ — การ์ด "รอรับใบกำกับ"/"เกินกำหนด" และปุ่ม
"ดูทั้งหมด" ในลิสต์ทั้งสอง พาไปหน้าบันทึกค่าใช้จ่ายพร้อม filter = รอรับ, การ์ด "ได้รับแล้ว" พาไปพร้อม
filter = ได้รับแล้ว, การ์ด "VAT ที่รอรับ" และ Quick Action "รายงานภาษีซื้อ" พาไปหน้ารายงานภาษีซื้อ,
Quick Action "เพิ่มค่าใช้จ่าย"/"Import Excel" พาไปหน้าบันทึกค่าใช้จ่ายพร้อมเปิดฟอร์ม/แผงนำเข้าให้ทันที
กลไกนี้ทำผ่าน `NavIntent` (type ใน `lib/navigation.ts`) ส่งเป็น optional prop ล้วน ๆ ไม่กระทบ
business logic ใด ๆ ทั้งสิ้น

### หน้า "บันทึกค่าใช้จ่าย"

หลังย้าย KPI Cards และสรุป VAT รายเดือนออกไป Dashboard แล้ว หน้านี้เหลือเฉพาะส่วนที่ใช้ทำงานจริง:
Search, Filter สถานะ, ปุ่มนำเข้าจาก Excel, ปุ่มเพิ่มรายการ, ตารางรายการค่าใช้จ่าย และ Pagination
(10 รายการต่อหน้า) — logic การกรอง/ค้นหา/เรียง/บันทึกข้อมูลทั้งหมดเหมือนเดิมทุกประการ ไม่มีการแก้ไข

> **หมายเหตุเรื่อง Route:** เว็บนี้ไม่มี URL แยกต่อเมนู — ทุกเมนูอยู่ใน URL เดียวคือ `/dashboard`
> แล้วสลับเนื้อหาด้วย state ฝั่ง client (จำเมนูที่เลือกไว้ใน localStorage) การเพิ่มเมนู "Dashboard"
> และเปลี่ยนหน้าแรกของระบบในรอบนี้จึงไม่ได้เพิ่ม/แก้ URL ใด ๆ เลย เป็นแค่เปลี่ยนค่าเริ่มต้นของ state
> ฝั่ง client (`DEFAULT_ACTIVE_ID` ใน `lib/navigation.ts`) จาก `record-expense` เป็น `dashboard`

พฤติกรรมอื่น ๆ: หมวด "บันทึกการจ่ายเงิน" และ "กระทบยอด" (รวมหมวดย่อย "VAT Reconcile" ที่ซ้อนอยู่
ข้างใน) ยุบ/ขยายได้อิสระจากกัน, ระบบจำเมนูที่เลือกและสถานะยุบ/ขยายไว้ในเบราว์เซอร์ (ข้าม refresh ได้
แต่เป็นการจำต่อเบราว์เซอร์ ไม่ใช่ต่อบัญชี), และเมื่อจอแคบกว่า 992px (มือถือ/แท็บเล็ต) sidebar จะซ่อน
เป็นค่าเริ่มต้น เปิดผ่านปุ่มแฮมเบอร์เกอร์มุมบนซ้าย ปิดได้ด้วยการแตะพื้นหลังมืด, ปุ่ม X, ปัดนิ้วไปทางซ้าย
บน sidebar, หรือเลือกเมนูใด ๆ

## 8. รายงานภาษีซื้อ (VAT Reconcile)

หน้า **"รายงานภาษีซื้อ"** (เมนู กระทบยอด > VAT Reconcile > รายงานภาษีซื้อ) สรุปใบกำกับภาษีซื้อที่
"ได้รับแล้ว" ทั้งหมด จัดรูปแบบใกล้เคียงรายงานภาษีซื้อของกรมสรรพากร ใช้ข้อมูลชุดเดียวกับหน้า
"บันทึกค่าใช้จ่าย" ทุกประการ (SWR cache เดียวกัน ผ่าน `INVOICES_SWR_KEY`) ไม่มีการเก็บข้อมูลซ้ำซ้อน

**ฟิลด์ใหม่ตอนกดปุ่ม "ได้รับแล้ว"** (นอกเหนือจากเลขที่ใบกำกับภาษี/วันที่ได้รับเอกสารเดิม):

| ฟิลด์ | บังคับ | ความหมาย |
|---|---|---|
| วันที่ใบกำกับภาษี | บังคับ | วันที่พิมพ์อยู่บนใบกำกับภาษีจริง — คนละค่ากับ "วันที่ได้รับเอกสาร" เดิม |
| เดือนที่ใช้เครดิต VAT | บังคับ | เดือนที่นำใบกำกับภาษีนี้ไปยื่น ภ.พ.30 จริง (ตั้งต้นเป็นเดือนปัจจุบัน แก้ได้) |
| ปีที่ใช้เครดิต VAT | บังคับ | ปี พ.ศ. ที่ใช้เครดิต (ตั้งต้นเป็นปีปัจจุบัน แก้ได้) |

**หลักการกรองรายงาน — สำคัญ:** ตัวกรองเดือน/ปีของหน้ารายงานอิงจาก **เดือน/ปีที่ใช้เครดิต VAT**
เท่านั้น ไม่ใช่วันที่ใบกำกับภาษีหรือวันที่ได้รับเอกสาร เพราะบริษัทอาจนำใบกำกับภาษีไปเครดิตคนละเดือนกับ
วันที่บนเอกสารก็ได้ ตัวอย่างเช่น ใบกำกับภาษีลงวันที่ 28 มิถุนายน แต่บริษัทได้รับเอกสารจริง 5 กรกฎาคม
แล้วนำไปยื่นเครดิตของเดือนกรกฎาคม รายการนี้จะปรากฏใน **รายงานเดือนกรกฎาคม** เท่านั้น (ทดสอบไว้ใน
`lib/vatReportLogic.test.ts` และ `e2e/purchaseTaxReport.spec.ts`)

คอลัมน์ในรายงาน: วันที่ใบกำกับภาษี, เลขที่ใบกำกับภาษี, ผู้ขาย, เลขประจำตัวผู้เสียภาษี, รายการ,
ฐานภาษี, VAT 7%, ยอดรวม พร้อมแถวสรุปยอดรวมท้ายตาราง — export เป็น **Excel** หรือ **PDF** ได้
(ปุ่มถูก disable ถ้าไม่มีรายการในช่วงที่เลือกอยู่)

> **หมายเหตุด้านเทคนิค (ฟอนต์ไทยใน PDF):** ฟอนต์มาตรฐานที่มากับ jsPDF (helvetica ฯลฯ) ไม่มีตัวอักษร
> ไทยเลย จึงฝังฟอนต์ **Sarabun** (ฟอนต์ที่ราชการไทยแนะนำให้ใช้ในเอกสารทางการ เทียบเท่า TH Sarabun New)
> แบบ base64 ไว้ใน `lib/pdfFonts.ts` โดยตรง (ที่มา: Google Fonts, สัญญาอนุญาต SIL Open Font License —
> ใช้/แจกจ่ายซ้ำได้โดยไม่มีค่าใช้จ่าย ตรวจสอบ glyph coverage ตัวอักษรไทยครบถ้วนแล้วด้วย fontTools
> ก่อนนำมาใช้งาน) ไฟล์นี้เป็นข้อมูลฟอนต์ล้วน ไม่ควรแก้ไขด้วยมือ — ถ้าต้องการเปลี่ยนฟอนต์ในอนาคตให้สร้าง
> ไฟล์ใหม่แทนที่ทั้งไฟล์

**สโคปที่ยังไม่ทำรอบนี้:** "รายงานภาษีขาย" (เมนูคู่กัน) ยังไม่มีฟีเจอร์จริง เพราะระบบปัจจุบันมีข้อมูล
ฝั่ง "ซื้อ" (ใบกำกับภาษีจากผู้ขาย) เท่านั้น ยังไม่มีระบบบันทึกยอดขาย/ข้อมูลลูกค้าเลย ต้องสร้างระบบบันทึก
ยอดขายก่อนจึงจะทำรายงานภาษีขายที่ใช้ข้อมูลจริงได้ (ตัดสินใจร่วมกับผู้ใช้แล้วว่าทำรายงานภาษีซื้อก่อน)

## 9. ACC Reconcile AI Copilot (ผู้ช่วย AI)

ปุ่มลอย (avatar วิดีโอแบบ chroma-key โปร่งใส) มุมล่างขวาของทุกหน้า ตั้งแต่หน้า **Login** เป็นต้นไป — กด
เปิดเป็นแผงแชทเลื่อนเข้าด้านข้าง ถามวิธีใช้งานหน้าต่างๆ ได้, สั่งให้พาไปหน้าที่ต้องการ ("ไปหน้ารายงานภาษีซื้อ"),
หรือสั่งให้ช่วยชี้ตำแหน่งปุ่ม/เมนูที่กำลังพูดถึง ("ปุ่มเพิ่มรายชื่ออยู่ตรงไหน") ได้จริง

**หลักการสำคัญ — ผู้ช่วยนี้ "แนะนำ" เท่านั้น ไม่เคยทำ action แทนผู้ใช้:** ความสามารถที่กดได้จริงมีแค่ 2 อย่าง
คือ **นำทาง** (สลับเมนูให้) กับ **ไฮไลต์** (เลื่อนจอไปหา+เรืองแสงชี้ element จริงที่มีอยู่บนหน้าปัจจุบัน) —
ไม่มีทางบันทึก/แก้ไข/ลบ/อนุมัติ/ส่งข้อมูลใดๆ แทนผู้ใช้ได้เลยแม้แต่ทางเดียว ทุกอย่างที่ต้องแก้ไขข้อมูลจริงยังคง
ต้องให้ผู้ใช้กดเองในหน้าปกติเสมอ

### Local Mode (ค่าเริ่มต้น — ไม่ต้องตั้งค่าอะไรเลย)

ผู้ช่วยเวอร์ชันที่ใช้งานได้ทันทีหลัง deploy โดยไม่ต้องตั้งค่าอะไรเพิ่มเติมเลย ตอบคำถามจากฐานความรู้ที่เขียนไว้
ในเครื่อง (`lib/assistantKnowledge.ts`) ครอบคลุมทุกหน้าจริงในระบบ — จับคู่คำถามด้วยการเทียบคำสำคัญล้วนๆ
(ไม่มีเครือข่าย ไม่มีค่าใช้จ่าย ไม่ต้องมี API key ใดๆ) ถ้าจับคู่ไม่ได้ ผู้ช่วยจะตอบข้อความสำรองที่แนะนำหัวข้อ
ที่ถามได้ตรงๆ แทนการเงียบหรือตอบมั่ว

### เปิดใช้ AI จริง (ทางเลือกเสริม)

ถ้าต้องการให้คำถามที่จับคู่กับฐานความรู้ในเครื่องไม่ได้ (เช่น คำถามกำกวม/ซับซ้อนกว่าที่เขียนไว้) ถูกส่งต่อไปให้
AI จริงตอบแทน ตั้งค่าตัวแปรใน `.env.local` (ดูตัวอย่างเต็มใน `.env.local.example`) 3 ตัวคู่กัน:

| ตัวแปร | ค่า |
|---|---|
| `NEXT_PUBLIC_ASSISTANT_REMOTE_ENABLED` | `true` |
| `ASSISTANT_PROVIDER` | `anthropic` หรือ `openai` |
| `ANTHROPIC_API_KEY` หรือ `OPENAI_API_KEY` | API key จริงของ provider ที่เลือก (server-only — ไม่มี prefix `NEXT_PUBLIC_` จึงไม่ถูกส่งไปฝั่ง client เด็ดขาด) |

เส้นทางนี้ตอบเป็น**ข้อความล้วนๆ เท่านั้น** ไม่เคยสั่งนำทาง/ไฮไลต์กลับมาเลย (ความสามารถ Smart Action ทั้งหมด
มาจากฝั่ง Local Mode เท่านั้น) และถ้าเรียกไม่สำเร็จไม่ว่ากรณีใด (ไม่ได้ตั้งค่า key, เครือข่ายพัง, response
ผิดพลาด) ระบบจะ fallback กลับข้อความสำรองของ Local Mode เงียบๆ เสมอ ไม่มีทาง error โผล่ให้ผู้ใช้เห็นในแชท

### หมายเหตุด้านเทคนิค

ไม่มีตาราง Supabase ใหม่/migration ใดๆ สำหรับฟีเจอร์นี้ทั้งสิ้น — เก็บแค่สถานะเปิด/ปิดแผง (`benz_assistant_open`)
ไว้ใน `localStorage` ของเบราว์เซอร์เท่านั้น ส่วนประวัติแชทเป็นแบบ in-memory ต่อ session (รีเฟรชหน้าแล้วหาย
ตามที่ตั้งใจ — มีปุ่ม "ล้างการสนทนา" ในแผงสำหรับเริ่มใหม่ระหว่าง session เดียวกันด้วย) `app/api/assistant/chat/route.ts`
เป็น Next.js Route Handler ตัวแรกของโปรเจกต์นี้ (ก่อนหน้านี้ทั้งระบบเป็น client-heavy เรียก Supabase ตรงๆ
ล้วนๆ) แต่ inert โดย default จนกว่าจะตั้งค่า AI จริงตามด้านบน
